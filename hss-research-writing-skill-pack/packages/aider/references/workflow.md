# Workflow guide for HSS research writing

这个文件是主技能的展开说明。只在需要更细流程时读取。

## 1. 任务启动：先写 brief，再碰正文

在 `00-task-brief.md` 里尽量明确：

- 研究问题
- 对象 / 时间 / 空间 / 案例范围
- 材料类型
- 研究方法或分析路径
- 目标输出
- 语言与引用风格
- 材料缺口
- 风险点（证据不足、页码不全、案例不对称、伦理不明等）

### brief 的最低格式

```md
# Task brief

## Question

## Scope

## Current claim boundary

## Materials on hand

## Missing materials

## Planned outputs

## Risks and unresolved items
```

## 2. 材料台账：先建总表，再谈观点

推荐先用 `templates/source-ledger.md` 建一个总表。所有材料只要进入工作台，就给一个固定 ID。

### ID 命名建议

- `L-001`：literature 二手文献
- `A-001`：archive 一手档案
- `I-001`：interview 访谈
- `F-001`：fieldnote 田野笔记
- `P-001`：policy / law / institutional text
- `M-001`：media / discourse material
- `D-001`：dataset / statistical source

### 台账至少要记录

- ID
- 类型
- 标题 / 描述
- 作者 / 机构 / 说话者
- 时间 / 年份
- 来源 / 链接 / 馆藏 / 出处
- 研究相关性（高 / 中 / 低）
- 可靠性与偏向说明
- 已读状态
- 是否提炼
- 是否被引用
- 是否完成核验

### 需要单独标出的风险列

- `source_risk`：来源可靠性问题
- `citation_gap`：引用元数据缺失
- `ethics_risk`：匿名、隐私、敏感信息风险
- `comparison_risk`：案例是否不可比

## 3. 单份材料卡片：把“读过”变成“能调用”

### 对二手文献至少提炼这些字段

- 研究问题
- 对象 / 时间 / 范围
- 方法 / 材料
- 核心论点
- 概念定义
- 对你的研究的价值
- 限制 / 盲点 / 争议点
- 可直接引用的句子与页码

### 对档案 / 一手材料至少提炼这些字段

- 生产者是谁
- 面向谁
- 何时、为何产生
- 所处制度 / 政治 / 组织语境
- 文本在说什么
- 文本避而不谈什么
- 是否有自我粉饰 / 立场性 / 审查痕迹
- 是否有可与其他来源互证的点

### 对访谈 / 田野至少提炼这些字段

- 访谈对象位置
- 发生场景
- 主题标签
- 情绪与犹豫点
- 典型句
- 偏离案例
- 研究者位置性提醒

### 对政策 / 法律 / 媒体至少提炼这些字段

- 文本类型
- 主要行动者
- 规范目标 / 政策工具 / 话语框架
- 执行机制
- 对象是谁
- 与其他文本的连续 / 断裂关系

## 4. 从材料到比较：不要只做摘要，要做对照

推荐至少建立一种对照文件：

- 时间线：同一议题的阶段变化
- 比较网格：不同案例 / 文本 / 作者的并列比较
- 编码表：访谈 / 田野 / 判决 / 政策的主题标签
- 概念表：同一概念在不同作者处的不同含义

### 时间线特别适合

- 政策演化
- 制度变迁
- 争论升级 / 断裂
- 媒体框架变化
- 历史阶段划分

### 比较网格特别适合

- 多案例研究
- 多文本比较
- 政策对照
- 不同理论路径对比

## 5. 证据矩阵：让主张真正落地

推荐使用 `templates/evidence-matrix.md`。

### 一个好证据矩阵至少回答五件事

1. *主张是什么*：一句话写清楚，不要超过两句。
2. *证据是什么*：具体到来源 ID。
3. *证据为什么支持它*：解释中间的推理桥。
4. *有什么反证或限制*：主动写出弱点。
5. *把握度是多少*：高 / 中 / 低。

### 证据矩阵的常见错误

- 只写“有文献支持”，没有具体来源
- 只写“有案例”，没有说明案例怎么支撑判断
- 把规范立场当成经验事实
- 把一个个案写成普遍机制
- 把“作者这样说”直接当成“现实如此”

## 6. 论证地图：先搭骨架再写肉

推荐使用 `templates/argument-map.md`。

### 论证地图至少包含

- 中心论点
- 2-4 个分论点
- 每个分论点对应的主要材料
- 潜在反驳点
- 你的回应路径
- 哪些部分仍需补材料

### 什么时候说明还不能写

如果出现下面任一情况，要告诉用户“可以先写提纲 / 备忘录，但不宜装成定稿”：

- 核心贡献句还说不清
- 中心论点还找不到足够证据
- 关键引语或页码都没落地
- 比较维度还不对称
- 反例太强但尚未处理
- 伦理或匿名问题未解决

## 7. 从论证地图到提纲

推荐使用 `templates/section-outline.md`。

### 提纲写法原则

- 先写每一节“要证明什么”，再写“放什么材料”
- 每节尽量只承担一个核心分析动作
- 标出哪些段落是：描述 / 解释 / 反驳 / 讨论 / 过渡
- 如果某节只能列材料，不能说清要证明什么，说明这节还没想透

## 8. 草稿阶段的最低交付物

即便用户想快速拿到草稿，也建议同时交付：

- 一个一句话贡献
- 一个当前证据充分度说明
- 一个待核清单
- 一个 next step

这样用户可以直接进入下一轮，而不是被“看起来很完整但内里空心”的草稿误导。

## 9. 修订日志

每次重写、补证或调整结构，建议写进 `80-revision-log.md`：

- 问题是什么
- 哪一段改了
- 用了哪些材料修补
- 还剩什么没解决

这是最容易被忽视、但最适合 CLI 工具长期协作的一步。
